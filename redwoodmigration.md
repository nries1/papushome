# Background
I just added redwoodjs to this repo because I want to offload the crud app portion of the app onto redwoodJS' react, graphql, and prisma stack. The problem is that redwood needs yarn, so I want to migrate my existing services to use yarn instead of npm. The pre-redwood services are in the server/ directory, robot-vision-publisher, and robot-vision-worker. And are all managed with docker-compose.yml

Database migration

### Diff 1

### Diff 2 (in progress)

Porting the rest of `server/` (everything outside `server/database/dao.ts`, which Diff 1 covered) file by file into the Redwood `api/` tree.

**Architecture decision: where does a persistent background service (not a GraphQL request) live in Redwood?**
- Redwood supports a custom `api/src/server.ts` (via `yarn rw setup server-file`, which scaffolds it and adds `@redwoodjs/api-server` as an explicit `api/package.json` dependency) — this is the officially-supported hook for running arbitrary startup code alongside the API process.
- **Caveat that changed the plan:** `api/src/server.ts` is only used by `yarn rw serve` (production). `yarn rw dev` uses its own dev-mode bootstrap (`nodemon` + `rw-api-server-watch`) and does **not** invoke `server.ts` at all — confirmed by grepping the CLI source (`server.ts` only referenced from `serve.js`/`serveApiHandler.js`, not `dev.js`).
- So the actual initialization hook used is a **side-effect import of `src/lib/mqtt` from `api/src/functions/graphql.ts`** — that file is loaded in both dev and serve/production (including under a custom `server.ts`, since `createServer()` still auto-discovers and mounts all functions), so it's the one place guaranteed to run in both modes. `server.ts` itself stays as the stock scaffolded template; it wasn't the actual trigger point here, but it's still needed for `yarn rw serve` to be a valid deploy target going forward.

**Done:**

  - **`device_logs` vs. deferred "logs" domain — these are different things.** The Diff 1 deferral was about *our own server's operational logging* (`app_logs`/`api_logs`, replaceable by Redwood's logger). `device_logs` is physical IoT device telemetry (ESP32s reporting their own log lines over MQTT) — real domain data that `deviceHealthStats.deviceLogErrors7d` already depends on. ### Diff 3 — Auth (2026-07-10)

Replaced the `api/src/lib/auth.ts` stub with a real port of the old `server/api/auth.ts` Express middleware.

**How it works:** Cloudflare Access sits in front of nginx/api and injects `cf-access-authenticated-user-email` once a user has authenticated via SSO — there's no login flow inside this app and nothing to decode (no JWT verification here, same trust boundary as the old middleware, which also just read the header raw). `ADMIN_EMAILS` (`nries1@gmail.com`, `avalongoebel@gmail.com`) gates admin-only actions, same list as before.

### Diff 5 — `getDevicesWithStatus` (2026-07-10)

Closes gap #1, the last item from the Diff 2 audit checkpoint. `devicesWithStatus.sdl.ts` + `api/src/services/devicesWithStatus/get.ts`, replacing the dashboard's big per-device aggregate behind `GET /api/devices/status`.

No backing Prisma model — it's a correlated-subquery join across five tables (`devices`, `rooms`, `device_presence`, `device_logs`, `environment_readings`, `tank_readings`, `watering_events`) plus tank-calibration math in `CASE` expressions, `json_object_agg`, and `row_to_json`. Stayed raw SQL via `db.$queryRaw`, same precedent as `deviceHealthStats`/`tankSensorMetrics` — this isn't a case where Prisma's query builder was being avoided out of habit, it's a genuine escape-hatch case (aggregation shapes and calibration math the query builder can't express).

**Verified raw-row JS types empirically rather than assuming from the SQL** (per the standing rule from the tank-sensor-metrics gotcha) — two things a naive reading of the query would get wrong:
- `jsonb` columns (`config`) and `json_object_agg`/`row_to_json` results (`latest_env_readings`, `latest_tank_reading`, `latest_pump_event`) come back auto-parsed into plain JS objects by the driver — no manual `JSON.parse` needed, and no `Prisma.Decimal` wrapping either, since the values already passed through a JSON serialize/deserialize round-trip server-side (unlike the tank-sensor-metrics case, where raw `numeric`/`double precision` columns came back Decimal-wrapped or not depending on the SQL function).
- **Timestamps behave differently depending on whether they pass through `row_to_json`.** Top-level timestamp columns (`last_boot`, `last_seen`, `latest_env_timestamp`) come back as real JS `Date` objects, same as any other Prisma raw-SQL timestamp column. But the `timestamp` fields *nested inside* `latest_tank_reading`/`latest_pump_event` — because they're serialized through `row_to_json` — come back as **ISO strings**, not `Date` objects (JSON has no native date type). The row type reflects this split explicitly (`Date | null` vs `string`) rather than typing everything the same. Redwood's `DateTime` GraphQL scalar accepts both transparently, so no extra conversion was needed — just got the raw-row type right so a future callsite doesn't assume `.getTime()` works on the nested ones and crash at runtime.
- `config`/`latest_env_readings` are typed `Prisma.JsonValue` (from `@prisma/client`), not `Record<string, unknown>` — the generated `JSON!` scalar type expects `JsonValue` specifically, and TypeScript's structural check on the recursive `JsonValue` union doesn't accept a plain `Record<string, unknown>` in its place.

Query name is `devicesWithStatus` (no `get` prefix), matching the plural-noun convention already used everywhere else (`devices`, `wateringEvents`, `latestTankReadings`, etc.) rather than the old function's `getDevicesWithStatus` name.

**Function-vs-Fastify-route interaction verified, not assumed:** confirmed the custom `/vision/stream` route coexists cleanly with Redwood's own function-discovery routing (`/:routeName` and `/:routeName/*`, which Fastify's router prioritizes as more-specific-wins over static paths regardless of registration order) — re-tested `/config` and a real GraphQL query against the same running `yarn rw serve` process immediately after the SSE route responded correctly, both still worked.

### Diff 7 — Logging decision, and closing out the migration (2026-07-10)

The logging strategy Diff 1 deferred on is now decided, and it turns out to resolve the deferral rather than extend it.

**The decision:** no DB-backed operational logging going forward. App-level warn/error events already moved to Redwood's built-in `logger.warn`/`logger.error` (pino → structured JSON on stdout, captured by `docker-compose logs`) consistently across every diff in this migration — that part wasn't really a pending decision by this point, just a pattern already in continuous use since `mqtt.ts` in Diff 2. **Patterns established (apply to future domains):**

- Service directories are **plural** (`api/src/services/devices`, not `device`) — canonical Redwood convention.
- SDL field names and nullability should mirror `schema.prisma` exactly. Don't write DTO mappers — for fields that genuinely diverge (relations, renames), use a field resolver: `export const <TypeName>: <TypeName>RelationResolvers = { ... }` in the service file. It's applied automatically to every query/mutation returning that type, not per-operation.
- Redwood's generated resolver map types (`QueryResolvers['x']`, `MutationResolvers['x']`) are optional properties, so indexing them directly yields `Fn | undefined` and breaks destructured-arg type inference in the editor. Always wrap: `NonNullable<QueryResolvers['x']>`.
- Offset pagination convention: `(page, rows, order)` args on list queries — `limit = rows ?? <domain default>`, `offset = page ? page * limit : 0`, `order` is `'ASC' | 'DESC'` (case-insensitive) mapped to Prisma's `orderBy` direction.
- Multiple `.sdl.ts` files can each declare their own `type Query { ... }` / `type Mutation { ... }` — Redwood merges them, so no need to centralize.
- For computed/aggregate results with no backing Prisma model (stats, health checks), use `db.$queryRaw(Prisma.sql\`...\`)` with `Prisma.sql`/`Prisma.empty` for optional filter fragments — same escape hatch as the pgvector workaround, just for aggregates Prisma's query builder can't express (`STDDEV_POP`, `PERCENTILE_CONT`, mode, etc.). Always verify raw-row column JS types with `SELECT pg_typeof(...)` against the actual query rather than assuming from the SQL function name — see the tank sensor metrics gotcha above.
- Don't reach for raw SQL just because the old `dao.ts` used it — a lot of the old raw SQL exists because Kysely has no JSON-path or relation-filter helpers, not because Postgres needs it. Check whether Prisma's query builder already covers it before writing `$queryRaw`: JSON field filtering (`{ path: [...], equals }`) and relation filtering (`{ room: { name } }`) replaced two raw-SQL conditions in `latestEnvironmentReading` with plain Prisma calls.

**Environment notes:**

- `yarn` isn't on `PATH` by default in this environment — run `corepack enable` (picks up the `packageManager: yarn@4.6.0` pin in `package.json`).
- Root `package.json` needed a manual `"gql": "yarn rw g types"` script — Redwood doesn't ship one out of the box.

# Known Issues

## Prisma + pgvector (`home_knowledge.embedding`)
The old db has one column that doesn't map cleanly to Prisma:

```sql
embedding vector(768)
-- with a partial HNSW index:
CREATE INDEX idx_home_knowledge_embedding ON home_knowledge
  USING hnsw (embedding vector_cosine_ops) WHERE (embedding IS NOT NULL);
```

Prisma has no native `vector` type and no way to declare an HNSW/partial index in schema.prisma. Workaround used:

- `datasource db` declares `extensions = [vector]` (requires the `postgresqlExtensions` preview feature on the generator) so Prisma emits `CREATE EXTENSION IF NOT EXISTS "vector";` in the migration automatically.
- The column itself is declared as `embedding Unsupported("vector(768)")?` on the `HomeKnowledge` model. Prisma will create/alter the column correctly, but the generated Prisma Client has **no typed accessor** for it — any read/write of `embedding` has to go through `$queryRaw`/`$executeRaw`.
- The HNSW index can't be expressed via `@@index` at all (no access-method or partial-index support in Prisma schema syntax), so after running `yarn rw prisma migrate dev --create-only`, the generated `migration.sql` needs a manual line added before applying:
  ```sql
  CREATE INDEX "idx_home_knowledge_embedding" ON "home_knowledge"
    USING hnsw ("embedding" vector_cosine_ops) WHERE ("embedding" IS NOT NULL);
  ```
- This is a one-time manual step per migration that touches this table/index — if `schema.prisma` changes again in a way that regenerates `home_knowledge`'s DDL (e.g. `prisma migrate dev` without `--create-only`, or `db push`), the HNSW index has to be re-added by hand again, since Prisma's migration diffing doesn't know about it.

### Upstream status (checked July 2026)

No native fix coming soon for the ORM version we're on — this workaround is the correct long-term approach, not a stopgap for a release that's about to land.

- **Stable Prisma ORM (v6.x/7.x):** no native vector type. [ORM 6.13.0](https://www.prisma.io/blog/orm-6-13-0-ci-cd-workflows-and-pgvector-for-prisma-postgres) added pgvector as an *enabled extension* for Prisma Postgres, but reading/writing vector columns still requires the same `Unsupported()` + custom migration + raw/TypedSQL approach used here. The post says native support is "coming soon" with no date attached.
- **Open tracking issues, nothing committed:**
  - [prisma/prisma#26546 "First class Vector support"](https://github.com/prisma/prisma/issues/26546) (Mar 2025) — open, no linked PR or milestone.
  - [prisma/prisma#18442 "Support for pg_vector"](https://github.com/prisma/prisma/issues/18442) (Mar 2023) — open, no assignee.
  - [prisma/prisma#28867](https://github.com/prisma/prisma/issues/28867) — a regression where Prisma 7.1.0 misdetects schema drift on `Unsupported("vector")` columns. Closed as a bug but no confirmed fix has landed — worth checking before upgrading the Prisma version pinned in this repo.
- **Where real work is happening:** not core Prisma ORM, but **Prisma Next** (the experimental successor, eventually Prisma 8), via its extension model. `@prisma-next/extension-pgvector` provides typed vector columns (`field.column(vector(1536))`) and a `cosineDistance()` query method compiling to the pgvector `<=>` operator. Early Access only — not something to build on yet, but worth revisiting once it graduates out of Early Access.
